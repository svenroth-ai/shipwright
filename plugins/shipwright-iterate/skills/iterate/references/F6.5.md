# F6.5 — Attach Commit SHA to F5b Event (SKIPPED in the worktree flow)

> **Normal worktree iterates SKIP this step.** Since
> iterate-2026-05-29-events-jsonl-worktree-commit the `work_completed` event is
> committed by **F6** inside the worktree (per-tree, PR-committed model). An
> in-place SHA patch *after* F6 would re-dirty the just-committed
> `shipwright_events.jsonl` and require a second commit — the opposite of the
> clean single-commit PR we want. So the event ships with `commit=""`; the
> commit↔run linkage lives in the F6 commit's `Run-ID:` footer and the event's
> `adr_id == run_id`. The F11 verifier accepts a committed event with an empty
> `commit` field.

The step below is **legacy / non-worktree only** — for a caller that records
the event into a tree it does NOT commit (e.g. an out-of-band replay), where
patching the SHA after the fact is still useful. It patches the event in place
(atomic temp + rename).

```bash
uv run "{shared_root}/scripts/tools/finalize_iterate.py" attach-commit \
  --project-root "{project_root}" \
  --event-id "{event_id from F5b}" \
  --commit "$(git -C "{project_root}" rev-parse HEAD)"
```

If `event-id` cannot be located, the script returns non-zero; investigate
(typical cause: F5b skipped or the event was rotated out of events.jsonl).
