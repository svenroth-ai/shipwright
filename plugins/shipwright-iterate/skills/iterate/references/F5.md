## Test Completeness Ledger block (small+)

At small / medium / large complexity, also write the
`iterate_latest.test_completeness` block — the machine-readable form of the
Step 7.5 ledger. The F11 verifier `check_test_completeness_ledger`
(`shared/scripts/tools/verifiers/iterate_checks.py`) reads it and STOPs the
run on any gap. At trivial complexity the gate is skipped; record
`{"status": "n/a", "justification": "..."}` so the F11 summary still shows a
line.

```json
"test_completeness": {
  "status": "complete",
  "behaviors": [
    { "behavior": "untested_testable>0 → FAIL", "disposition": "tested",
      "evidence": "test_verify_iterate_finalization.py::test_...positive PASSED" },
    { "behavior": "send live email", "disposition": "untestable",
      "reason_code": "requires-prod-credential",
      "reason": "needs the real SMTP secret, absent from CI" }
  ],
  "counts": { "testable": N, "tested": N, "untestable": M, "untested_testable": 0 },
  "enumeration_basis": { "acs": N, "covered_acs": N }
}
```

Rules the verifier enforces (fail-closed, ERROR):

- `status` ∈ {`complete`, `n/a`}. `n/a` is valid only at **small** (with a
  non-empty `justification`) or trivial (auto). At **medium/large** `n/a` is
  rejected — a medium+ iterate has testable behavior by definition, so it
  MUST enumerate (`status=complete`).
- `status=complete` needs ≥ 1 behavior; each `disposition` ∈ {`tested`,
  `untestable`}. **No "deferred" / "untested" / "acceptable"** — that escape
  hatch is exactly what the gate kills.
- `tested` rows must cite `evidence`; `untestable` rows must cite a
  `reason_code` from the closed vocabulary (see
  `references/confidence-anti-patterns.md`).
- `counts.untested_testable` must be `0`.
- `enumeration_basis` (optional but recommended at medium+): if `acs >
  covered_acs`, the gate fails — guards a vacuous pass via under-enumeration.
- **`category` (optional per behavior)** — a behavior may carry
  `"category": "integration"` to mark it as INTEGRATION coverage (a real-scenario
  test proving cross-component pieces compose). At **medium+**, when the iterate
  touches FRAMEWORK cross-component machinery (the `cross_component` risk flag), the
  F11 verifier `check_integration_coverage` RECOMPUTES the flag from the diff and
  STOPs unless ≥ 1 behavior is `category:"integration"`. Reference pattern:
  `shared/tests/test_parallel_merge_cascade_integration.py`. See SKILL.md Risk
  Taxonomy + `references/confidence-anti-patterns.md` (Integration Stopping Rule).

## Per-test execution evidence (traceability R1 / G5)

`iterate_latest` above is the **aggregate** per-layer count. It is layer-blind
about *which* test proved *which* requirement, and a silent `test.skip` can leave
it green. For requirement→test traceability, finalization also emits **per-test**
execution evidence so `coverage[layer]=ok` can require a tagged test that is
enabled AND executed=pass (not merely present).

Drop this run's raw runner reports under
`.shipwright/compliance/evidence/` (`junit.xml`, `playwright.json`,
`vitest.json`) — the compliance `test_links` collector calls the
`execution_evidence` reader (`_execution_evidence_io.refresh_index`) on regen to
normalize them into `.shipwright/compliance/test-evidence-index.json` and join
`status`/`executed` onto each manifest test link. The reader is **fail-closed**:
a missing report resolves to `not_run` (never a self-reported pass), a
green-but-skipped required-layer test is `MISSING`, and an out-of-vocab evidence
value is coerced, not trusted. A layer that genuinely cannot run in an
environment needs an explicit **expiring waiver** (honored while valid, fails
when expired) — never self-reported success.

**Emit-side (TT5) — clear before the run, stage after.** Use the shared
`evidence_drop` CLI so the enforcing **cross-layer F11 gate** reads fresh,
HEAD-attributable evidence (it trusts the index ONLY when a provenance sidecar
proves this run staged those reports). Before F0/F0.5 runs the tests, clear stale
evidence; after they produce reports, stage them:

```bash
# once, before this run's test execution (F0/F0.5) — never inherit a prior run's reports:
uv run shared/scripts/lib/evidence_drop.py clear --project-root .
# after the runners produce reports (F0.5/F5) — records run_id + head_commit provenance:
uv run shared/scripts/lib/evidence_drop.py stage --project-root . \
  --run-id {run_id} --head-commit "$(git rev-parse HEAD)" \
  --junit <path> --playwright <path> --vitest <path>
```

Order matters: `clear` (before tests) → run tests → `stage` (copies reports + provenance).
You do NOT rebuild the index by hand — the cross-layer gate REGENERATES it from the staged
raw reports itself (the R3 "regenerate, don't trust" rule applied to evidence), and the F5
compliance regen also rebuilds it. The gate trusts the result ONLY when the provenance
run_id matches this run AND the recorded `head_commit` is an ancestor of / equal to the
commit being verified — so a stale, planted, or foreign-branch index can never credit a
pass. Without `--head-commit` the evidence is not HEAD-attributable and is rejected.

> **Staging is REQUIRED (not optional prose) for a behaviour-changing medium+ iterate.**
> The cross-layer gate is fail-closed: a required layer with no staged executed-passing
> evidence reads `MISSING` and BLOCKS. So if this iterate changes an FR's behaviour, you
> MUST `stage` the run's passing reports (with `--head-commit`), and the report paths must be
> **project-root-relative** so the test ids join the manifest (the A6 per-runner-`bases`
> normalization is a documented deferral). A pure-tooling / no-FR-delta iterate needs no
> staging (the gate does not fire).

> **Why the next steps run BEFORE the commit (F6):** the finalization
> script and the `iterate_history` append write artifact files tracked
> in the repo. Running them after F6 would dirty the working tree
> immediately after committing and force a `git commit --amend` —
> which conflicts with the "never amend" rule in the global
> `CLAUDE.md`.
