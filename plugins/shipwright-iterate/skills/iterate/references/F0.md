# F0 — Fresh Verification Gate

**Leak-guard first.** Confirm the run is still isolated —
`{project_root}` is an iterate worktree and nothing leaked into the
main repo working tree:

```bash
uv run "{shared_root}/scripts/checks/check_iterate_isolation.py" \
  --project-root "{project_root}" --run-id "{run_id}" --stage f0
```

Non-zero exit = **STOP**: either the run is not executing in a
worktree, or it wrote into the main tree (compared against the B1a
Step-1 snapshot). Investigate and revert before continuing.

Then re-run the full suite NOW — do not rely on earlier results.

## Which command

**If the project declares a `suite` block in `shipwright_test_config.json`**
(this monorepo does), run the canonical suite runner:

```bash
uv run "{shared_root}/scripts/tools/run_test_suite.py" --project-root "{project_root}"
```

**Otherwise** (no `suite` block — the default for an adopted project), run the
project's own full-suite command as before, e.g.:

```bash
npx vitest run
npx tsc --noEmit

# Integration tests (if CRUD/DB changes)
npx vitest run --config vitest.integration.config.ts

# pgTAP tests (if new RLS migrations)
supabase test db
```

The runner is **opt-in**: with no `suite` block it refuses with an actionable
message and changes nothing for an existing project.

## What the runner does (and deliberately does not)

Since `iterate-2026-07-14-f0-parallel-suite`. The suite's units are already
isolated processes by design (ADR-044 namespace collision), so they are the
natural parallelisation unit — they were merely being run one after another
(measured: ~9.8 min serial, ~1.9 min now).

- **Units are DISCOVERED, not hardcoded** — the same selection rule as `ci.yml`
  (every `plugins/*/` with `pyproject.toml` + `tests/`, the three `shared/` test
  dirs, `integration-tests/`). A new plugin is picked up automatically; a
  hardcoded list would silently stop testing it. Guarded by
  `shared/scripts/tools/tests/test_f0_ci_parity.py`.
- **`pytest-xdist` is a per-unit OPT-IN** (`suite.xdist` allowlist), never a
  global `-n auto`. **`shipwright-compliance` is deliberately absent from the
  allowlist and must stay absent** — it is measurably NOT xdist-safe (5
  shared-state races in `test_test_evidence.py` under `-n 8`; the same tests
  pass 37/37 serially).
- **"Did pytest run?" is PROVEN, not guessed.** Each unit writes a JUnit report;
  the file exists iff pytest executed. This matters because `uv run` *also* exits
  `1` when it merely fails to build the environment — and sniffing its prose is
  unsound (pytest pluralises `error` → `errors`, so a fixture-level race that
  ERRORs every test reads nothing like a normal failure). `rc 1` + report = a real
  test failure; `rc 1` + no report = an infrastructure fault.
- **Serial re-verify (no false STOP).** A unit that reports a genuine pytest *test
  failure* is re-run **serially, without xdist, alone, after the pool drains**, in
  a clean temp dir — the way F0 used to run it. That verdict is authoritative.
  Red-in-parallel + green-alone does **not** stop the gate; the runner warns and
  the unit must be triaged (it is a race **or** a flaky test — the runner does not
  claim to know which).
- **An infra fault is retried once with the IDENTICAL command shape** (xdist still
  on). A *deterministic* fault (rc `5` nothing-collected, a usage error, an
  unprovisionable xdist) reproduces and still fails the gate — nothing is
  laundered. A *transient* one recovers: 18 concurrent `uv` processes create
  contention (hardlink races in the shared uv cache) that serial runs never had,
  and refusing those a retry would simply trade a race-induced false STOP for an
  infra-induced one. **The retry must never strip xdist** — that would green a
  suite that never ran the fan-out its config demands.
- The outer process pool and the inner xdist workers draw from **one** CPU budget
  (`cpu_count - 2`, or `suite.max_workers`), so the machine is not oversubscribed.
  A hang is capped by `suite.timeout_seconds` (default 1800) and becomes a fault,
  not an infinite silent block.

> **F0 is an accelerated PRE-gate — be honest about it.** The retries remove false
> STOPs. They do *not* prove serial equivalence for units that *passed*: a test that
> passes only *because* of parallelism would not be caught here. That is why **CI
> stays serial** (`.github/workflows/ci.yml` is deliberately NOT parallelised) — CI
> is the authoritative serial gate before merge. This is not a convention but a
> **guard**: `test_f0_ci_parity.py::test_ci_stays_SERIAL` fails if `ci.yml` ever
> gains `-n auto` / `--numprocesses` / `pytest-xdist`. Do not "fix" CI by
> parallelising it, and do not claim F0's verdict is provably identical to a serial
> run.
>
> The retry gets a clean **temp dir** — but the repo **working tree is shared** by
> all units and is not reset. A unit reddened by another unit's tree pollution will
> reproduce on its retry and stop the gate. That is a known limit, not a claim.

## Read the actual output

- Exit code is 0
- Every unit `PASS` (a `RACE` warning is green, but must be followed up)
- No type errors

If ANY unit fails: **STOP.** Go back to the build step and fix before
continuing. Do not proceed to F1 with failing tests.
