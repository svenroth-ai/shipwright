# F11 — Push Branch, Open PR & Verify

**Leak-guard first** — confirm the run never touched the main repo
working tree (snapshot-and-diff against the B1a Step-1 snapshot):

```bash
uv run "{shared_root}/scripts/checks/check_iterate_isolation.py" \
  --project-root "{project_root}" --run-id "{run_id}" --stage f11
```

Non-zero exit = **STOP**.

**Refresh-if-behind guard (Option A — serial integrate, never server-side).**
GitHub auto-merge does a server-side 3-way merge and NEVER runs the
regenerate-at-merge resolver, so a branch that fell behind `origin/<default>`
while open would merge **stale** (Group-E staleness) or stall **DIRTY** on the
regenerated snapshots. Bring the branch current through `ensure_current.py`
FIRST (fetch → if behind: merge → churn-reconcile → regenerate snapshots →
commit) so the PR arms auto-merge from a current, already-regenerated tree. A
branch that is already current is a clean no-op (nothing fetched-merged-committed).

```bash
guard=$(uv run "{shared_root}/scripts/tools/ensure_current.py" \
  --project-root "{project_root}" --run-id "{run_id}" \
  --reason "F11 pre-merge refresh: {run_id}") || {
    echo "$guard"
    echo "STOP: ensure_current failed (non-churn/source conflict?) — resolve by hand, then re-run F11."
    exit 1
  }
echo "$guard"
```

This runs BEFORE the push below, so the single `git push -u` ships any integrate
commits. A non-churn (source) conflict exits non-zero — the same hard safety
gate as the resolver — and **STOPS** the run.

**Push the iterate branch and open a PR.** An iterate run NEVER checks
out, merges, or pushes the default branch — that races every other
parallel iterate. One iterate = one branch = one PR.

```bash
default_branch=$(git -C "{project_root}" symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's@^refs/remotes/origin/@@' || echo main)
git -C "{project_root}" push -u origin "iterate/{slug}"
# gh has no `-C` analog — it acts on the cwd's repo. cd into the worktree
# so gh can never operate on the main repo even if the shell cwd drifted.
cd "{project_root}"
gh pr create --base "$default_branch" --head "iterate/{slug}" \
  --title "<type>(<scope>): <description>" \
  --body "<summary>

## Empirical Test Completeness
<derived from iterate_latest.test_completeness — e.g.>
Testable: N · Tested: N · Untestable (structural reason): M · Untested-but-testable: 0 ✅

Run-ID: {run_id}"

# --- Arm GitHub-native auto-merge (B4.5 Phase 3) -----------------------
# An iterate PR squash-merges itself once the Required Checks pass. The
# arm is gated THREE ways so it can never run away:
#   (1) branch scope — ONLY `iterate/*` arms. A manual, human-authored
#       PR never reaches this prose, and the `case` guard means it would
#       not self-arm even if it did (opt in per-PR with your own
#       `gh pr merge --auto` or an `automerge` label instead).
#   (2) fail-soft — if "Allow auto-merge" / branch protection is OFF in
#       repo settings (or the branch can't merge), `gh pr merge --auto`
#       errors. We WARN and leave the PR open. NEVER `|| exit`: a missing
#       repo setting must not break F11 for every future iterate.
#   (3) campaign-defer — an `--autonomous` campaign exports
#       SHIPWRIGHT_ITERATE_AUTOMERGE=0 so sub-iterate PRs do NOT arm here;
#       the orchestrator merges each PR in turn, interleaved-serial
#       (references/campaign-mode.md) — only ONE campaign PR is open at a
#       time, so the auto-merge × per-PR-snapshot × up-to-date cascade can't form.
head_branch="iterate/{slug}"
pr_url=$(gh pr view "$head_branch" --json url -q .url)
if [ "${SHIPWRIGHT_ITERATE_AUTOMERGE:-1}" = "0" ]; then
  echo "Auto-merge arm DEFERRED (SHIPWRIGHT_ITERATE_AUTOMERGE=0) — campaign orchestrator merges this PR in turn (interleaved-serial): $pr_url"
else
  case "$head_branch" in
    iterate/*)
      gh pr merge "$pr_url" --auto --squash --delete-branch \
        || echo "WARN: auto-merge not armed (repo 'Allow auto-merge' off / no branch protection / conflict) — PR left open for manual merge: $pr_url" ;;
    *)
      echo "Non-iterate branch '$head_branch' — auto-merge NOT armed (manual opt-in only)." ;;
  esac
fi
```

**Empirical Test Completeness line (always, small+).** Render one line from
`iterate_latest.test_completeness` into the PR body so the reviewer reads the
answer to *"did you test everything testable?"* without asking. `Untested-but-testable`
is `0` by construction — the F11 verifier (below) fails closed otherwise.
At trivial complexity emit `Completeness: n/a — <justification>`.

If `iterate/{slug}` diverged from `origin/<default>` since B1a fetched it
(another iterate merged meanwhile), the **refresh-if-behind guard above**
(`ensure_current.py`) already merged + regenerated the derived snapshots — a
**merge**, not a rebase, because only the merge path runs the regenerate-at-merge
resolver (a rebase replays your commits but never re-derives the churn snapshots,
so it would merge stale). That guard is the deliberate, single conflict point.
**This run never merges the PR itself** — GitHub squash-merges it once the
Required Checks go green (the arm above). But **the iterate is NOT done when the
PR is armed — it is done only when the PR is _delivered_: MERGED with all Required
Checks GREEN**, confirmed by the **Delivery Watch** below. Arming and walking away
is "shoot and forget"; a check can fail afterward and the PR sits BLOCKED. If the
arm can't be set (repo "Allow auto-merge" off, branch protection absent, a merge
conflict, or `SHIPWRIGHT_ITERATE_AUTOMERGE=0` under a campaign) the PR
stays open for the campaign orchestrator to merge in turn, or a manual, human-gated
merge — surface that explicitly (**not delivered yet**), never claim "done".

**Gate check:** F5b records the `work_completed` event into **this
worktree's own** `shipwright_events.jsonl` (the per-tree, PR-committed model —
`events_log.resolve_events_path` returns the literal `{project_root}` copy),
and F6 stages it so it ships in the PR. The deterministic verifier below checks
this authoritatively via `check_events_has_commit`, which — when the log is
tracked — asserts the event is in the **committed HEAD blob**, not merely the
working copy (AC4). So the event MUST be at `{project_root}/shipwright_events.jsonl`
AND committed by F6; if F6 forgot to `git add` it, the verifier fails closed.

**Deterministic verifier.** After the gate check, run the
finalization verifier and fail the iterate run on red:

```bash
uv run "{shared_root}/scripts/tools/verify_iterate_finalization.py" \
  --run-id "{run_id}" \
  --project-root "{project_root}" \
  --commit "$(git -C "{project_root}" rev-parse HEAD)"
```

Exit 0 = green (or warnings only), exit 1 = at least one required
artifact missing. Add `--strict` to treat warnings as errors.

This run includes `check_test_completeness_ledger` (small+): it fails closed
when any behavior in `iterate_latest.test_completeness` is testable-but-untested,
an `untestable` row lacks a closed-vocabulary `reason_code`, or the
enumeration falls short of the AC count. That is the gate that makes the
operator's pre-merge "did you test enough?" question unnecessary.

It also includes `check_review_record` (small+; skipped at trivial): the run's
`.shipwright/planning/iterate/{run_id}/reviews.json` must exist and hold NO
review type still `pending`. All five — `self` · `plan` · `code` · `doubt` ·
`external_code` — are materialized up front and each is closed by the pass that
owns it; a pass that did not run is closed `not_run` / `not_applicable` with a
`--disposition` naming the rule (a bare "skipped" is rejected). Fails closed on a
missing, unreadable, or schema-invalid record, and on one whose `run_id` names a
different run. Remediation is printed with the exact command per outstanding
type; a run that predates the record closes everything at once with
`record_review_pass.py close-missing`. The point of the gate is that an empty
Review row in the Mission view always means "genuinely not run", never "nobody
wrote it down".

It also includes `check_iterate_no_direct_decision_log`: an iterate whose commit
modified `.shipwright/agent_docs/decision_log.md` fails closed — record the ADR as
an F3 decision-DROP (`write_decision_drop.py`), never append to `decision_log.md`
directly (the `ADR-NNN` is assigned at `/shipwright-changelog` release). Paired
with a stricter `check_adr_in_iterate_history` that accepts a run-id ADR identity
only when the drop actually carries the ADR or a `Run-ID:` line is in
`decision_log.md`.

**Delivery Watch — the iterate is DELIVERED only when the PR is MERGED + green.**
NEVER arm auto-merge (or push) and declare "done": that is "shoot and forget" —
a Required Check can fail afterward and the PR sits BLOCKED, un-merged, red. Watch
the PR to a terminal state:

```bash
pr_url=$(gh pr view "iterate/{slug}" --json url -q .url)
uv run "{shared_root}/scripts/tools/watch_pr_delivery.py" --pr "$pr_url"
# Fail CLOSED: ONLY exit 0 (merged + green) is "delivered". Every other outcome
# STOPS the run (exit 1) — never fall through and claim "done".
case "$?" in
  0) echo "DELIVERED — PR merged + all Required Checks green." ;;
  2) echo "NOT DELIVERED — a Required Check FAILED. Diagnose (gh run view --log-failed <run>), FIX, re-push, then re-run this watch. The iterate is NOT done." ; exit 1 ;;
  3) echo "NOT DELIVERED — PR was CLOSED unmerged." ; exit 1 ;;
  4) echo "NOT DELIVERED — still PENDING at the timeout. Re-run the watch to keep waiting; do NOT claim 'done'." ; exit 1 ;;
  *) echo "NOT DELIVERED — delivery watch could not read PR state (gh error). Verify + re-run before claiming delivery." ; exit 1 ;;
esac
```

`watch_pr_delivery.py` polls `gh pr view --json state,mergeStateStatus,statusCheckRollup`
until the PR is `merged` (exit 0 = delivered), a Required Check fails (exit 2 =
**STOP, not delivered** — fix + re-push + re-watch), the PR is closed (3), or the
poll times out while still pending (4 = keep watching, NOT done). A `needs:`-skipped
required job (Tier-1/2 `PR Review`) counts as a pass, not a failure. **Under a
campaign (`SHIPWRIGHT_ITERATE_AUTOMERGE=0`)** the orchestrator merges each PR in
turn, interleaved-serial (campaign-mode.md step 3g); the same "delivered = merged"
bar applies per sub-iterate. For plugin/`shared/*` changes, "delivered to the
runtime" additionally requires the post-merge `update-marketplace.sh` cache sync.
