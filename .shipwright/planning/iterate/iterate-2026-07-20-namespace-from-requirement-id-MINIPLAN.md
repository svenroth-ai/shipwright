# Mini-plan — S3: derive the manifest namespace from the requirement id (schema 2 → 3)

## Problem

`Requirement.namespace` was whatever the caller passed, and every caller passed the
spec's parent DIRECTORY name (`Path(spec).parent.name`). So every manifest key
(`01-adopted::FR-01.01`) was hostage to a directory rename: rename a split, and all 15
requirement keys move, silently.

## Change

- `Requirement.namespace` is now a read-only **property** derived from the id's group
  digits (`FR-01.03` → `01`). It is no longer a constructor field, so no call site can
  hand in a path. `parse_requirements` loses its `namespace=` parameter.
- Manifest `schema_version` 2 → 3. The v3 JSON-schema pins requirement keys to
  `^\d{2}::FR-\d{2}\.\d{2}$`, so a reintroduced path-derived namespace fails write-time
  validation — the structural enforcement of the AC, not a prose rule.
- A frozen contract fixture (`tests/contracts/test-traceability-3.0.json`) plus a gate
  that reads the baseline from `origin/main` via `contract_baseline.frozen_fixture_diff`.

## Verified, not assumed

The live manifest was keyed `01-adopted::FR-01.01`. All 15 keys move to `01::FR-01.01`.
Regenerating the real monorepo manifest and comparing node-by-node against the committed
one: **zero inner-field differences** across `id`, `tests`, `coverage`, `required_layers`,
`required_layers_source`, `title`, `status`, `priority`, `spec_path`. Only the keys and
`schema_version` moved.

## The consequence the spec did NOT anticipate — please review this hardest

A v3 key is a **pure function of the id**. Under v2 two specs declaring `FR-03.01` got
two distinct keys (`01-a::`, `02-b::`) and therefore two manifest nodes. Under v3 they
claim ONE key. A plain `dict[key] = req` resolves that last-writer-wins, which would:

1. silently DELETE a requirement from the manifest, and
2. silently disarm the fan-out ambiguity guard — `_group_d_manifest.collision_ids` and
   `_rtm_layer_columns` detect a shared display id by COUNTING the nodes carrying it.
   One node ⇒ no count ⇒ no ambiguity ⇒ the fanned `ok` becomes creditable again. That
   is exactly the false-green a prior Codex CRITICAL review closed (MUST-FIX 2).

Mitigation built here: the collector keeps the FIRST claimant (discovery is sorted, so
deterministic) and records every later one in a new, omitted-when-empty
`duplicate_requirement_keys` field. `ambiguous_ids()` unions those ids back into the
collision set, so the guard survives the collapse. A regression test asserts the collapse
really happens AND that the RTM still renders `?`.

**Questions for the reviewer:** is reporting-and-keeping-first the right resolution, or
should a duplicate FR id fail the regen closed? Is unioning into the existing collision
set sufficient, or does some third consumer count nodes independently?

## Also changed (consumers that would have broken silently)

- `_rtm_layer_columns.load_layer_index` hard-gated on `schema_version != 2` and would
  have returned an EMPTY index — every RTM layer column silently `—`.
- `layer_cells(index, split, fr_id)` rebuilt the lookup key from the split DIRECTORY.
  The `split` parameter is removed; the key comes from the id.
- `_group_d_manifest.load_manifest` had the same `!= 2` gate → D-layer/D-orphan SKIP.

Both keep exactly-one-version strictness (a stale v2 manifest degrades to
SKIP/`—` until the next regen), mirroring how a v1 manifest was already treated.

## Deliberately NOT changed

adopt's decisions/resolutions map (`find_ambiguous_frs` → `"app::FR-03.02"`) stays
split-name namespaced. It is a separate, user-authored contract that never touches the
traceability manifest; changing it would break existing predeclared decision files. The
two key forms now differ, which is noted in the test that exercises both.

## Risks

- Cross-repo: `test-traceability.json` is generated by monorepo tooling but committed in
  the WebUI repo, so the WebUI sees a whole-file diff on its next regeneration. Expected.
- The WebUI's `readTraceabilityIndex` never reads the composite key (consumption is
  `Object.values(...)` → inner `.id`), which is what makes the key change safe.
