# Iterate: Admit `ci-security.json` to the churn allowlist

- **Run ID:** iterate-2026-07-15-churn-allowlist-ci-security
- **Intent:** bug (classification defect in the churn-merge resolver)
- **Complexity:** medium (floor: `cross_component` — the diff touches
  `churn_merge.py`, framework merge machinery, so the F11 verifier
  `check_integration_coverage` recomputes the flag and demands a
  `category:"integration"` behavior)
- **Risk flags:** `cross_component`
- **Spec Impact:** NONE — framework-internal merge machinery; no project FR /
  `spec.md` / product surface changes. (BUG default; behaviour-correcting.)

## Problem

`shared/scripts/lib/churn_merge.py` defines

```python
DERIVED_MDS      = COMPLIANCE_MDS | AGENT_DOC_MDS          # .md files only
CHURN_ALLOWLIST  = DERIVED_MDS | {EVENTS_LOG, TEST_RESULTS, TRIAGE_LOG}
```

`.shipwright/compliance/ci-security.json` is a **derived compliance snapshot**
(the structured CI-security summary the WebUI dashboard reads), regenerated by
the *same* single-producer path as the five compliance `.md` files. But because
`DERIVED_MDS` enumerates only `.md` files, the `.json` snapshot is **absent from
`CHURN_ALLOWLIST`**. `classify()` therefore returns it as `blocking`, so
`complete_merge` returns `"blocked"` and `integrate_main.integrate` /
`ensure_current` `git merge --abort` whenever `ci-security.json` conflicts —
which it does whenever two branches each regenerated it. Hit while merging #374.

### Root cause (empirically reproduced)

```
$ python -c "from lib.churn_merge import classify; print(classify(['.shipwright/compliance/ci-security.json']))"
([], ['.shipwright/compliance/ci-security.json'])   # resolvable=[], blocking=[ci-security.json]
```

The exclusion is an **oversight of the `.md`-shaped allowlist**, not a
deliberate block: `ci-security.json` is regenerable and its exclusion has no
protective purpose (contrast `architecture.md`, which is *curated prose*
deliberately kept out so it reaches a human).

## Fix (classification + rollback parity)

Admit `.shipwright/compliance/ci-security.json` to `CHURN_ALLOWLIST` via a named
`CI_SECURITY_SUMMARY` constant. It then falls into `complete_merge`'s catch-all
`else` branch and is resolved exactly like a derived MD: `--theirs` placeholder
at conflict, then regenerated-from-the-merged-tree in the follow-up.

**Rollback parity (added per external review, GPT-5.4 finding 2).** Because the
file is now allowlisted, `integrate_main`'s transactional restore-on-regenerate-
failure path must also restore it — otherwise a best-effort `refresh_ci_security`
that mutated the file *before* another generator errored would leave a modified
tracked file after a "failed, undone" merge. Fix is narrow: the restore set
becomes `sorted(DERIVED_MDS | {CI_SECURITY_SUMMARY})` (no `DERIVED_MDS` rename).

**Why no code beyond the allowlist is needed (verified):**

- `complete_merge`'s `else` branch already handles *any* allowlisted
  non-log path via `--theirs` (placeholder) + stage. `ci-security.json` routes
  there automatically.
- `regenerate_tracked_snapshots` → `finalize_iterate._update_compliance` runs
  `update_compliance.py`, which regenerates the **whole** `.shipwright/compliance/`
  dir (returns every file via `iterdir`) and best-effort-refreshes
  `ci-security.json` (`refresh_ci_security`, fail-soft — leaves the summary
  untouched when there is no fresh `security.yml` run). So the follow-up leaves
  a clean tree; no dirty/unstaged snapshot is left behind.
- `--theirs` (origin/main's summary) is the correct placeholder: `ci-security.json`
  mirrors the latest *mainline* CI-security scan, so the more-current side wins —
  identical treatment to the derived MDs.

### Alternative considered

Fold `ci-security.json` into `DERIVED_MDS` (rename → `DERIVED_SNAPSHOTS`) so it
also joins the rollback set. Rejected: `DERIVED_MDS` is `.md`-named and imported
by name across three modules — renaming is a refactor far beyond the defect. The
rollback-parity concern it was meant to solve is instead closed narrowly by
`DERIVED_MDS | {CI_SECURITY_SUMMARY}` at the one restore site (above), leaving
`DERIVED_MDS` and its `.md` semantics untouched.

- `*_config.json` / compliance-snapshot IO boundary — but this change does **not**

- Framework cross-component merge machinery (`churn_merge` ↔ `resolve_churn_conflicts`
  ↔ `integrate_main`/`ensure_current`) — the composition axis under test.

## Acceptance Criteria

1. `.shipwright/compliance/ci-security.json` ∈ `CHURN_ALLOWLIST`.
2. `classify(['.shipwright/compliance/ci-security.json'])` returns it as
   **resolvable**, not blocking.
3. The `docs/hooks-and-pipeline.md` "Merge reconciliation of churn artifacts"
   table documents the `ci-security.json` row with a **regenerate** strategy, and
   the bidirectional 1:1 registry test (`test_churn_merge_doc_sync`) stays green.
4. A real-git parallel-merge cascade with a conflicting `ci-security.json`
   resolves through `integrate_main.integrate` **without blocking**
   (`status == "ok"`, no conflict markers) — the `cross_component`
   `category:"integration"` behavior.
5. On a regenerate-failure, `integrate_main.integrate` restores
   `ci-security.json` to the merge commit (`--theirs` mainline placeholder), so a
   `regenerate_failed` merge leaves a clean tree (rollback parity).

## External review disposition (GPT-5.4 + Gemini 3.1 Pro)

Gemini: no findings. GPT-5.4: 6 findings, dispositioned:

- **F1 (medium, merge-direction):** verified — `integrate` always merges
  `origin/<default>` *into* the iterate branch, `ensure_current` delegates to it,
  so `--theirs` is always mainline. AC-5 test asserts the placeholder is the
  mainline side.
- **F2 (medium, rollback asymmetry):** ADOPTED — rollback set now includes
  `CI_SECURITY_SUMMARY` (see Fix / AC-5). Chosen by the user over the strict
  pure-classification option.
- **F3 (medium, "regenerate" is best-effort):** the doc-table strategy cell is
  made precise — regenerate = best-effort refresh from the latest `security.yml`
  run, else the mainline `--theirs` placeholder stands.
- **F4 (low, doc-sync test):** `test_churn_merge_doc_sync` run explicitly;
  `regenerate` is an accepted strategy keyword (verified).
- **F5 (low, comment drift):** the `resolve_churn_conflicts.py` `else`-comment is
  phrased generically ("a regenerated allowlisted snapshot"), not per-file.
- **F6 (low, staleness):** pre-existing property of the fail-soft refresh design
  (the summary carries `generated_at`/run metadata); admitting one exact path to
  the allowlist introduces no new injection/authz surface. Out of scope.

## Internal code review disposition (shipwright-build:code-reviewer, opus)

No blocking findings — "correct, surgical, and well-tested; safe to merge." Two
non-blocking notes:

- **CR-1 (low, pre-existing forward-staging gap):** `regenerate_tracked_snapshots`
  stages only its `out` set (the derived MDs + agent MDs + campaign statuses),
  never `ci-security.json`. So if a *fresh* `security.yml` run rewrote
  `ci-security.json` during regen, that rewrite would be left unstaged and the
  follow-up commit (staged-only) would return `ok` with a modified-but-unstaged
  file. **This is present on EVERY integrate's regen step regardless of this
  change** (the reviewer confirmed it is independent of the allowlist admission);
  the common no-fresh-scan case is a clean no-op (verified). The pushed branch is
  always correct (it carries the committed `--theirs` placeholder); only a
  transient local dirty line + a lost local refresh result. Out of scope: closing
  it means staging `ci-security.json` in `regenerate_tracked_snapshots`, which
  ratchets the ADR-099 300-LOC exception on `resolve_churn_conflicts.py` (357→) and
  exceeds the approved scope. Documented here for pickup.
- **CR-2 (nit, test fidelity):** ADDRESSED — the cascade regen stub now mirrors
  production (regenerates the MD, leaves `ci-security.json` fail-soft-untouched),
  and the test asserts the mainline `--theirs` placeholder stands (`critical:1`),
  not the stub's own bytes.
