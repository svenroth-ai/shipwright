# `compliance/` -> `.shipwright/compliance/` Migration

Sub-Iterate F of the compliance relocation flipped this artifact to
`status: migrated` in `shared/scripts/lib/artifact_migrations.py`. This
turns the SessionStart drift detector into a **hard gate** for any
project that still has a top-level `compliance/` directory.

## What changed

- All Shipwright plugins now write compliance artifacts (dashboard.md,
  traceability-matrix.md, test-evidence.md, change-history.md, sbom.md,
  audit-report.md, security-scan-report.md, compliance_overrides.log,
  skill-compliance/) to `.shipwright/compliance/` instead of `compliance/`
  at the project root.
- Reads use only the canonical path -- there is **no Grace fallback**.
- The drift detector (`shared/scripts/hooks/check_artifact_drift.py`)
  scans for legacy `compliance/` directories at SessionStart. With
  `status: migrated`, finding one exits non-zero with a structured JSON
  payload that AI orchestrators parse.
- `.shipwright/` is an organizational umbrella (similar to `.git/`,
  `.vscode/`), not a security boundary. CLAUDE.md `@.shipwright/...`
  references resolve correctly. IDEs may hide the directory by default;
  toggle "show hidden files" if you need to browse it.

## Audit-relevant: compliance reports

The compliance directory holds **audit evidence**: SBOMs, test-evidence
reports, traceability matrices, and override logs. After migration:

- If you upload these to an external audit system (governance dashboard,
  Notion, SharePoint, etc.), update the source paths to
  `.shipwright/compliance/...`.
- The append-only override log (`compliance_overrides.log`) preserves
  its format and content; only its location changes. Existing log lines
  remain attributable to past timestamps after `git mv`.
- The skill-compliance subdir (`compliance/skill-compliance/`) preserves
  its `<phase>-<run_id>-<session_id>.json` layout under the new root.

## What you need to do (existing projects with `compliance/` at the root)

### Option 1 -- automated (recommended)

```bash
uv run shared/scripts/tools/migrate_artifact_dir.py \
  --artifact compliance \
  --project-root /path/to/your/project
```

This:
1. Verifies `compliance/` exists and `.shipwright/compliance/` is empty
   (or absent).
2. Uses `git mv` if the legacy directory is git-tracked, falling back
   to `shutil.move` otherwise.
3. Refuses non-destructively if either precondition fails -- nothing
   moves until you resolve the conflict manually.
4. Pass `--dry-run` first to preview, then re-run without it to commit
   the move.

After the move, commit:

```bash
git add -A
git commit -m "refactor: relocate compliance/ to .shipwright/compliance/"
```

### Option 2 -- manual

```bash
# From the project root:
mkdir -p .shipwright
git mv compliance .shipwright/compliance  # if git-tracked
# OR
mv compliance .shipwright/compliance      # if untracked

git add -A
git commit -m "refactor: relocate compliance/ to .shipwright/compliance/"
```

## Wenn `compliance/` Files in Git getrackt sind

`git status` zeigt nach dem Move noch die alten Files als deleted/added
an. `git mv` (Option 1 / 2 oben) handhabt das automatisch. Falls aber
Files manuell verschoben wurden:

```bash
git rm -r --cached compliance/   # unstage alte Tracking
git add .shipwright/compliance/  # neue Files stagen
git commit -m "refactor: relocate compliance/ to .shipwright/compliance/"
```

## CI/CD: Update Your Workflows

If your project has GitHub Actions, GitLab pipelines, deployment
scripts, audit-system upload jobs, or custom hooks that read from
`compliance/`, those will break after migration.

Search your repo:

```bash
grep -rn "compliance/" .github/workflows/ .gitlab-ci.yml scripts/ deploy/ 2>/dev/null
```

Common patterns to update:
- Wiki uploads of `compliance/dashboard.md` ->
  `.shipwright/compliance/dashboard.md`
- Audit-evidence archives in CI containing
  `compliance/{traceability-matrix,sbom,test-evidence}.md`
- Slack/Notion/SharePoint webhooks that excerpt
  `compliance/audit-report.md`
- External governance systems referencing `compliance/sbom.md` or
  `compliance/security-scan-report.md` URLs

## Don't Run Migration with Active Shipwright Sessions

Close any Claude Code sessions, IDE plugins, or background workers that
may write to `compliance/` (e.g. `compliance_overrides.log`,
`skill-compliance/<phase>-<run>-<session>.json`) before running
`migrate_artifact_dir.py`. The CLI does not currently take a
project-level lock; concurrent writes can produce split-brain artifacts.

## What the drift detector does on a stale legacy folder

If a SessionStart hook fires in a project where `compliance/` still
exists at the top level (and `status` for the migration is `migrated`):

1. The hook exits 1.
2. It writes structured JSON to stdout containing the finding plus a
   ready-to-paste `git mv` remediation list. Example:

   ```json
   {
     "ok": false,
     "drift": [
       {
         "artifact": "compliance",
         "legacy": "/abs/path/compliance",
         "canonical": "/abs/path/.shipwright/compliance",
         "remediation": "git mv compliance .shipwright/compliance"
       }
     ]
   }
   ```

3. It also writes `.shipwright/stale-folders.md` with the same data so
   you see it in your editor.

The hook never touches your files -- it only reports.

## What if I see `.shipwright/stale-folders.md`?

That file is the human-readable summary of any drift findings. It's
auto-deleted as soon as the next SessionStart runs cleanly. If it
persists, follow the remediation steps inside the file (or in the JSON
the hook printed to stdout).

## If migration fails

The CLI is non-destructive: if it refuses (canonical not empty, legacy
missing, both present, etc.) nothing on disk has changed. Common cases:

- **Both `compliance/` and `.shipwright/compliance/` exist:** the CLI
  refuses. Manually inspect both, decide which content is authoritative,
  remove the other (or merge), then re-run. For audit-evidence files:
  prefer keeping the newer set (audit reports are time-stamped; the
  older set can usually be regenerated via `update_compliance.py`).
- **`compliance/` is a symlink:** the CLI may refuse. Resolve the
  symlink target manually first.
- **CLI errored mid-move:** `git mv` and `shutil.move` are atomic on
  same-filesystem. Cross-filesystem `shutil.move` does copy-then-delete
  -- if interrupted, you may have both directories partially populated.
  Re-run the CLI; it will refuse and surface the half-state, then
  manual cleanup is required.

## Rollback

The migration is non-destructive (git history preserves both states).
To roll back to the legacy layout for a single project:

```bash
git mv .shipwright/compliance compliance
```

To roll back the framework-level decision (re-enable warn-only mode):
revert the status flip in `shared/scripts/lib/artifact_migrations.py`
back to `"in_progress"`. The drift detector will resume warn-only
behavior on the next SessionStart.
